the canvas

everything lives on an infinite scrollable world. the canvas is the outer viewport element; the world is what actually moves inside it. all frame positions are stored in world-space coordinates — so when you pan and zoom, the X/Y numbers in the inspector don't change, only your perspective on them does. the world's transform is just a CSS translate(x, y) scale(s) on a single div.

panning

three ways to get around:

  • Ctrl + drag on any empty area of the canvas
  • middle-click drag anywhere — canvas, world, or even on a frame (frames pass middle-click up to the pan handler)
  • Ctrl + drag from the pin button on a frame header also pans instead of starting a connection
panning works in both edit and view mode. when active, the cursor switches to grabbing and pointer capture is set on the drag start element — so the pan doesn't break if you accidentally drift outside the canvas boundary.

zooming

scroll wheel anywhere on the canvas zooms. the zoom is cursor-centered — whatever's under your mouse stays pinned while everything else scales around it. range is locked to 0.15×–2.5×. the zoom level is displayed live in the topbar readout.

zoom to fit

the zoom to fit button (or F) computes the bounding box of all frames and fits the camera to it with 80px padding. caps out at 1.4× so a single small frame doesn't get comically huge. no frames on board → falls back to centering.

reset view

reset view button or 0. drops the camera to world origin at 1× zoom, centered in the viewport. useful when you've lost your frames somewhere in the void.

the grid

the canvas background is a layered CSS grid: major lines at 160px intervals, minor at 32px. both scale and translate with the view so they stay anchored to world-space rather than the screen. it's a subtle orientation aid, not a real snap grid — for actual snapping see snap-align.

world origin orb

three concentric circles at world coordinate (0, 0), rendered in the back SVG layer. mostly decorative — gives you a visual anchor when the board is empty. it's also a valid snap target for Ctrl+drag alignment, so you can snap frame edges to the true origin if you're building something orderly.

frames

frames are the core unit. each one is an absolutely-positioned element in the world with a title, color, notes, body content (blocks), and a place in the sequence. internally they're called "cards" in the code but "frames" in the UI — same thing, don't let it trip you up.

creating

hit New Frame in the topbar or press N. new frames spawn at the center of your current viewport (screen → world conversion), cascading by 22px per frame (mod 6) so they don't all perfectly stack. the title input is auto-focused 30ms after creation so you can just start typing. default size: 280 × 210px.

moving

drag anywhere on the frame that isn't an interactive element (buttons, inputs, block controls, resize handle). position updates live; connections attached to the frame follow in real-time. hold Ctrl while dragging for snap alignment — see snap-align.

resizing

grab the resize handle at the bottom-right corner. drag to resize. minimums are enforced at 180px wide and 130px tall. the inspector's W/H fields update live while you drag.

deleting

  • Delete Frame button at the bottom of the inspector
  • Del or Backspace (when focus isn't in an editable input)
  • the × button in the frame's row in the left panel

deleting a frame immediately removes all connections that touch it. if multiple frames are selected, any of the above deletes the whole selection.

frame anatomy

visual structure (top → bottom)

  • color tag — thin accent strip at the top edge
  • header — number badge · title input · pin button
  • body — blocks, or an empty-state hint
  • toolbar — Text / Image / Link add buttons (edit mode only)
  • resize handle — bottom-right corner (edit mode only)

per-card CSS custom properties

  • --card-color — full hex value
  • --card-color-dim — hex at 40% opacity
  • --card-color-mid — hex at 70% opacity
  • --card-color-glow — hex at 22% opacity

available on every .card element if you want to use them in custom theme CSS.

frame numbers

the two-digit badge in the header reflects the frame's index in state.cards[], not a stored ID. numbers can shift when frames are deleted. the actual stable identifier is the internal uid() string (e.g. id3x9kmf2). the number is display-only.

frame line

a tagging property: left · right · up · down · none. set in the inspector, stored in card.frameLine, exposed as data-frame-line on the DOM element. loom doesn't enforce any visual behavior from it — it's an annotation for your own narrative direction system or CSS-based filtering. defaults to left.

blocks

a frame's body is made of zero or more blocks — the actual content units inside a frame. three types exist.

text block

added via the Aa toolbar button. it's a contenteditable div. type directly into it. changes save on each keystroke (with debouncing for history, see undo/redo). not editable in view mode.

image block

added via the ▢ button. triggers a hidden file picker (accept="image/*"). the image is read as a base64 data URL via FileReader and stored directly in the save state JSON. no external hosting, no compression — just raw base64. this means image-heavy boards bloat fast.

link block

added via the 🔗 button. starts as a temporary link-edit state with a text input. paste a URL, press Enter or click "Add". if you omit the protocol, https:// gets prepended automatically. the hostname is extracted and used as the display label. Escape cancels and removes the block. once confirmed, renders as a styled anchor (↗ icon, hostname as title, full URL as subtitle) that opens in a new tab.

reordering blocks

each block has a drag handle (the three-line icon on the block controls). drag it up or down within the frame body to reorder. the order is written back to card.blocks[] on pointer-up. a history entry is pushed if anything moved.

deleting blocks

the × button on the block controls removes it from the array and re-renders the frame. edit mode only.

toolbar and block controls are completely absent in view mode. the body becomes read-only. empty frames show a mode-appropriate hint string — encouraging in edit mode, neutral in view mode.

string connections

frames can be connected with a literal string — a physics-animated quadratic Bézier thread rendered in a dedicated SVG layer. connections form a directed acyclic graph: each frame can have at most one parent, cycles are forbidden.

creating a connection

drag from the pin button (the small tinted circle in the frame header's top-right corner) toward another frame and release over it. while you drag:

  • the string belly is physics-simulated — spring (k = 0.09), gravity (1.5), damping (0.86) — and lags behind your cursor in a satisfying way
  • hovering a valid target turns the string teal and highlights the target
  • hovering an invalid target (already has a parent, or would cycle back) turns the string red and highlights the target red
  • releasing over a valid target creates the connection and plays a settle animation (easeOutBack, 380ms) where the belly springs to its resting position
each frame can only have one incoming connection (one parent). trying to connect to a frame that already has a parent is an invalid drop and gets rejected with a toast message.

the thread rendering

each connection renders as four SVG paths stacked on the same Bézier d value:

  • hit path — transparent, 16px stroke, catches all pointer events
  • shadow path — dark, 3px, gives the string depth against the canvas
  • thread path — amber-gold rgba(228,190,112,0.9), 1.7px, with a soft SVG glow filter
  • shimmer path — very faint dashed highlight, 0.7px, adds texture

plus a tack icon at each endpoint — a three-layer circle SVG that looks like a physical thumbtack. the tack at the destination animates briefly with a "pop" class on the settle animation.

the resting belly position follows midpoint + dist × 0.28 + 36 — so longer strings sag more, shorter strings still have a gentle curve.

deleting a connection

click anywhere on the thread to delete it immediately. the hit path is 16px wide and transparent, so you don't need to be pixel-precise. deleted in view mode? nothing happens — click is guarded. want more options? right-click instead.

connection context menu

right-click a thread for a small context menu:

  • Send behind frames — sets conn.layer = "back", moves the string to the back SVG layer (z-index: -1)
  • Bring in front of frames — sets conn.layer = "front", front SVG layer (z-index: 50)
  • Apply to all strings — pushes the current connection's layer setting to every connection on the board at once

the active option is shown as checked and disabled (can't "bring in front" something already in front). dismiss by clicking elsewhere or pressing Escape.

SVG layer architecture

there are two SVG elements in the world: #conn-svg-back (behind frames, also holds the world origin orb) and #conn-svg (in front of everything). each has its own copies of the SVG filter defs with suffix variants (-back) to avoid ID collisions. during drag and settle animation, connections always render in the front layer regardless of their conn.layer setting.

selection

single selection

click a frame header, or click its row in the left panel frame list. the inspector updates to show that frame. the selected frame and its list row both get the .selected class.

marquee / box selection

drag from an empty area of the canvas (not on any frame). a selection rectangle appears. on mouse-up, any frames whose screen-space bounding rectangles intersect the box get selected. the marquee element is div.selection-box absolutely positioned within the canvas viewport.

hold Shift while drawing the marquee to add to existing selection instead of replacing it.

multi-selection

when 2+ frames are selected:

  • the inspector panel shows a count + lock message instead of properties
  • dragging any selected frame moves all selected frames together (group drag)
  • Del or the delete frame button removes all selected frames and their connections
  • per-frame properties (color, frame line, etc.) are locked until you're back to a single selection

press Escape to clear selection entirely.

group drag

triggered automatically when you drag any frame while 2+ are selected. all frames move by the same world-space delta. the frame you clicked is the anchor — if you hold Ctrl, snap-align is computed relative to the anchor and the same snap offset is applied to all others.

snap-align

hold Ctrl while dragging a frame (single or group) and its edges will snap to the edges and centers of all other frames, as well as the world origin.

snap candidates

for every non-selected frame, loom collects six values: x, x + w/2, x + w (left, center, right) and y, y + h/2, y + h (top, center, bottom). world origin (0, 0) is always included. your dragged frame's own six edge/center values are compared against all candidates. closest pair within the threshold wins — independently for X and Y.

threshold

8 screen pixels, divided by the current zoom scale to get world units. at 50% zoom that's 16 world units of pull; at 200% it's 4. the closer you zoom in, the more precise you need to be to trigger a snap.

guide lines

when an X or Y snap is active, a dashed guide line draws across the full visible viewport in the accent color at 0.85 opacity — vertical line for X snaps, horizontal for Y. they live in the front SVG layer and disappear the moment you release the mouse (the SVG is cleared on drag-up and guides aren't redrawn without an active snap).

snap is only active while Ctrl is held. let go mid-drag and the frame goes back to free movement immediately. the guide lines vanish with it.

inspector panel

the right panel. shows frame properties when exactly one frame is selected. shows an empty state when nothing is selected, or a lock message when multiple frames are selected. all inputs are disabled in view mode and during multi-select.

Delete Frame button

always at the bottom of the inspector content area. deletes the selected frame (or all selected frames if somehow multi is shown). same behavior as Del. can't be used in view mode — the button is inside the inspector which is hidden/locked anyway.

sequence / frame list

the left panel's main section. shows all frames in order, with the parent-child hierarchy from string connections rendered as indented tree nodes.

ordering & hierarchy

root frames (no parent) appear in state.cards[] creation order. each root's children are listed immediately after it, indented with a ↳ connector icon, then their children, and so on. depth is tracked and capped at CSS class depth-3 (deeper nesting still works, just doesn't indent further).

a visited-set prevents duplicate rendering in edge cases. any frames not reached through the root traversal (shouldn't happen since cycles are blocked) are appended at the end.

clicking a row

selects the frame and pans the camera to center on it. the pan is immediate (no animation). great for navigating large boards without hunting visually.

per-row elements

  • color dot — matches the frame's tag color
  • number — zero-padded two-digit sequence index
  • title — frame title, or "Untitled frame" if blank
  • × button — deletes that frame (only visible in edit mode)

soft updates

when you type in a frame's title input, the frame list updates the title text in-place via renderFrameListSoft() — patching just the text node instead of rebuilding the whole list. avoids flicker and preserves scroll position.

two modes, switchable from the dropdown in the topbar center. the dropdown also lives in the topbar and shows the current mode with a colored dot indicator.

persistence

the active mode is saved to localStorage under loom-mode-v1 and restored on page load. importing a JSON board always switches to view mode afterward — sensible default when someone hands you a finished storyboard.

body class

document.body gets the class view-mode in view mode. your theme CSS can use this selector to hide/show or restyle anything you want per-mode without touching the JS.

undo / redo

snapshot-based history. every meaningful action pushes a deep clone of the full state object (via JSON.parse(JSON.stringify(state))) onto a stack capped at 60 entries. when you undo/redo, the snapshot replaces the current state and the board re-renders entirely.

what's preserved vs. excluded

snapshots capture: cards, connections, projectName, nextNum. camera position (view) and panel widths (layout) are deliberately excluded — they're restored from the live state during undo/redo so the viewport doesn't jump around while you're correcting mistakes.

debounced text input

typing in title, notes, or text block inputs uses a debounced push with a 700ms delay. this means you don't get a history entry per keystroke — only after you stop typing for 700ms does the change commit. if you then do something destructive (delete a frame, add a connection, clear the board), any pending debounced push is flushed immediately as its own separate history entry first. the text change and the action end up as two distinct undo steps, which is what you want.

keyboard shortcuts

Ctrl+Z — undo    Ctrl+Y — redo    Ctrl+Shift+Z — redo (alternate)

undo/redo buttons are injected into the topbar (right of "New Frame") on init. they auto-disable when the stack boundaries are hit and are fully locked in view mode.

undo history is session-only. refreshing the page starts fresh. importing a board clears history and treats the import as the baseline. the initial loaded state always gets pushed as the first history entry, so you can undo back to it.

project management

project name

editable text field in the top of the left panel. max 60 chars. updates document.title in real-time as [name] — Loom. on blur, if the field is blank or whitespace, it resets to "Untitled Storyboard". locked in view mode.

export

downloads the full state object as formatted JSON. filename is slugified from the project name — spaces become hyphens, special characters stripped, .json extension appended. example: "My Film Act 2" → My-Film-Act-2.json. exported via object URL + anchor click trick, no server involved.

exported JSON shape: { cards, connections, view, layout, nextNum, projectName }.

import

triggers a JSON file picker. on load:

  • parses and validates (must have a cards array)
  • merges over DEFAULT_STATE() so missing fields get sane defaults
  • normalizes all cards and connections
  • clears undo history, clears selection
  • zooms to fit, saves to localStorage
  • automatically switches to view mode
  • pushes the imported state as the first (and only) history entry

invalid file → error toast, nothing changes.

clear

shows a native confirm() dialog — because this wipes everything. if confirmed, removes all cards and connections, clears selection, pushes to history, saves. edit mode only; does nothing if the board is already empty. it is undoable via Ctrl+Z (the confirm dialog says so).

panel resizing

both sidepanels have a thin drag handle at their outer edge. drag it to resize. the canvas area always keeps a minimum of 420px — if both panels combined would steal more than viewport - 420px, the right panel shrinks first, then the left.

mobile

at ≤ 680px viewport width, the left panel is hidden entirely and the right panel is fixed at 230px. resize handles are hidden. panel widths in state.layout are preserved but not applied until you're back on a wider screen.

persistence

panel widths are stored in state.layout and included in the localStorage save. normalized on every load (clamped to valid ranges), so a panel width that made sense at one screen size won't cause layout breakage at another.

persistence

everything auto-saves to localStorage on every meaningful change. no manual save button — it's always already saved.

keycontents
loom-storyboard-v1 full state JSON — cards, connections, view, layout, nextNum, projectName
loom-mode-v1 "edit" or "view" — the last active mode

normalization on load

the stored JSON is parsed and every field is sanitized before use: missing fields get defaults from DEFAULT_STATE(), invalid frameLine values reset to "left", invalid connection layer values reset to "front", layout widths are clamped to valid ranges. corrupted or missing JSON just starts fresh.

the image size problem

image blocks store raw base64 data URLs. a single uncompressed PNG can easily be 1–3MB. localStorage has a 5MB limit (varies by browser). hit it and you'll see a "could not save — storage may be full" toast, and subsequent saves are silently dropped. for boards with images, export early and often.

export as backup strategy

the exported JSON is the exact same shape as what's in localStorage. it's fully re-importable. there's no versioning layer beyond the storage key suffix (-v1), so if a future version of loom adds new fields, old exports will still load — Object.assign(DEFAULT_STATE(), parsed) fills gaps with sensible defaults.
